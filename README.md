
# Большие данные и распределённые системы  
## Лабораторный проект: учёт студентов по специальностям

## Описание проекта

Проект реализует распределённую информационную систему по варианту:

**Ведётся учёт студентов по специальностям. Имеется возможность принять студента на определённую специальность, отчислить за неуспеваемость, перевести со специальности на специальность. Сформировать отчёт о том, сколько студентов и на каких специальностях в данный момент обучаются, а также сколько было отчислено.**

Система развивалась по лабораторным работам **ЛР1–ЛР7** и в итоговом состоянии включает:

- PostgreSQL как основную БД
- основной REST/UI сервис на **FastAPI**
- отдельный сервис отчётов
- объектное хранилище **MinIO**
- брокер сообщений **RabbitMQ**
- клиентский многопоточный benchmark для ЛР7
- запуск всех компонентов через `docker compose`

---

## Что реализовано по лабораторным работам

### ЛР1. Развёртывание БД в Docker
Реализовано:
- контейнер с PostgreSQL
- volume для постоянного хранения данных
- инициализация схемы БД через `db/init.sql`

### ЛР2. CRUD-сервис
Реализовано приложение на FastAPI с REST API:
- `GET /` — список студентов
- `GET /{uuid}` — получение студента по UUID
- `POST /` — создание студента
- `PUT /{uuid}` — обновление студента
- `DELETE /{uuid}` — удаление студента

Дополнительно:
- Swagger/OpenAPI
- HTML UI для работы со студентами
- сидирование начальными данными через JSON seed

### ЛР3. Сервис отчётов
Добавлен отдельный `report_service`, формирующий отчёт по данным БД:
- сколько студентов обучаются
- на каких специальностях обучаются
- сколько студентов отчислено

### ЛР4. Изолирование доступа к сервису отчётов
Прямой внешний доступ к сервису отчётов был ограничен.
Отчёт получался через основной сервис как через шлюз.

### ЛР5. Файловое хранилище
Добавлен **MinIO**.
Логика отчёта была изменена:
- сервис отчётов строит отчёт
- сохраняет JSON-файл в MinIO
- основной сервис получает идентификатор файла
- скачивает файл из MinIO
- возвращает отчёт пользователю

### ЛР6. Брокер сообщений
Добавлен **RabbitMQ**.
Логика формирования отчёта была переведена на обмен сообщениями:
- основной сервис отправляет запрос на отчёт в очередь
- `report_service` слушает очередь и строит отчёт
- файл сохраняется в MinIO
- ответ о готовности отчёта возвращается через очередь
- основной сервис асинхронно ожидает ответ и скачивает файл из MinIO

### ЛР7. Параллельная обработка данных
Добавлен отдельный многопоточный benchmark:
- параллельное чтение
- создание
- обновление
- построение отчёта
- смешанные сценарии
- сохранение результатов в `txt`, `csv`, `json`
- построение графиков latency / throughput / error rate

## Итоговая архитектура

Финальная архитектура проекта:

- **db** — PostgreSQL
- **app** — основной сервис управления студентами и пользовательский интерфейс
- **report_service** — сервис формирования отчётов
- **rabbitmq** — брокер сообщений
- **minio** — внутреннее объектное хранилище файлов отчётов
- **pgadmin** — инструмент для просмотра БД
- **benchmark** — клиентский стенд нагрузочного тестирования

### Логика формирования отчёта в итоговой версии
1. Пользователь открывает вкладку «Отчёт» в основном UI.
2. Основной сервис публикует запрос в RabbitMQ.
3. Сервис отчётов получает сообщение из очереди.
4. Сервис отчётов строит отчёт по PostgreSQL.
5. Отчёт сохраняется в MinIO как JSON-файл.
6. Сервис отчётов отправляет сообщение о готовности отчёта обратно в RabbitMQ.
7. Основной сервис получает ответ, скачивает файл из MinIO и отдаёт данные в UI.

## Модель данных

### Таблица `specialties`
Содержит список специальностей.

Поля:
- `id`
- `name`

### Таблица `students`
Содержит сведения о студентах.

Поля:
- `id` — UUID
- `name` — ФИО студента
- `status` — статус (`учится`, `отчислен`)
- `specialty_id` — ссылка на специальность

---

## Seed-данные

В проекте используется минимальный JSON seed-файл:

`app/seed/students_seed.json`

Он нужен для того, чтобы после чистого старта контейнеров система уже содержала стартовые данные и можно было сразу:
- открыть UI
- получить отчёт
- запускать benchmark

Логика загрузки:
- при старте `app` проверяется таблица `students`
- если таблица пуста, данные загружаются из seed-файла

## REST API основного сервиса

### Основные CRUD endpoints
- `GET /` — список студентов
- `GET /{uuid}` — получение одного студента
- `POST /` — создание студента
- `PUT /{uuid}` — обновление студента
- `DELETE /{uuid}` — удаление студента

### Служебные endpoints
- `GET /health` — проверка состояния сервиса
- `GET /ui` — HTML интерфейс

### Endpoints отчёта
- `GET /report/json` — получение отчёта в JSON
- `GET /report/file` — скачивание файла отчёта

## Пользовательский интерфейс

В основном сервисе реализован интерфейс `/ui` с двумя вкладками:

### 1. Управление студентами
Позволяет:
- добавлять студентов
- изменять статус
- переводить между специальностями
- удалять записи

### 2. Отчёт
Позволяет:
- сформировать отчёт
- посмотреть итоговые показатели
- увидеть распределение обучающихся по специальностям
- скачать JSON-файл отчёта

---

## Формат отчёта

Отчёт содержит:
- общее количество студентов
- количество обучающихся
- количество отчисленных
- список специальностей с числом обучающихся на каждой

Пример структуры:

```json
{
  "total_students": 6,
  "active_students": 5,
  "expelled_students": 1,
  "by_specialty": [
    {
      "specialty_id": 1,
      "specialty_name": "Программная инженерия",
      "active_students_count": 2
    }
  ]
}
```

## Docker Compose

Проект запускается одной командой:

```bash
docker compose up -d --build
```

### Важные особенности итоговой конфигурации
- PostgreSQL инициализируется через `init.sql`
- БД не публикуется наружу
- `app` публикуется наружу на порт `8000`
- `pgadmin` публикуется наружу на порт `5050`
- `rabbitmq` публикуется наружу на:
  - `5672`
  - `15672`
- `report_service` не публикуется наружу
- `minio` в итоговой версии ЛР6–ЛР7 не публикуется наружу

## Benchmark (ЛР7)

Для нагрузочного тестирования используется отдельный каталог:

```text
benchmark/
```

### Возможности benchmark
Сценарии:
- `read_only`
- `create_only`
- `update_only`
- `report_only`
- `read_update_mix`
- `update_report_mix`

### Что считает benchmark
- общее число запросов
- число потоков
- суммарное время теста
- среднее время ответа
- медиану
- стандартное отклонение
- 90/95/99 квантиль
- пропускную способность
- процент ошибок

### Что сохраняется
- `benchmark_raw_*.csv`
- `benchmark_summary_*.csv`
- `benchmark_summary_*.json`
- `benchmark_report_*.txt`
- графики `.png`

### Запуск benchmark
```bash
cd benchmark
pip install -r requirements.txt
python benchmark.py
```

### Примеры запуска
```bash
python benchmark.py --requests 300
python benchmark.py --threads 1,2,4,8,16
python benchmark.py --scenarios report_only,update_report_mix
python benchmark.py --warmup 30
```

---

## Какой сценарий является самым тяжёлым

Наиболее тяжёлые сценарии для данной архитектуры:
- `report_only`
- `update_report_mix`

Потому что они нагружают всю распределённую цепочку:
- основной сервис
- RabbitMQ
- сервис отчётов
- MinIO
